# 性能优化篇：让自定义屏幕稳定、流畅、省资源

[上一课](12-中文点阵天气屏幕.md) · [返回总目录](../README.md) · [接口速查](07-Canvas接口速查.md)

性能优化不是为了把代码写得更“高级”，而是为了让屏幕在真实设备上长期稳定运行。OmniWatch 自定义样式运行在资源有限的设备上，屏幕刷新、数据传输、Python 执行和内存分配都会消耗时间。样式越复杂，越需要主动控制绘制范围、数据规模和每帧计算量。

这一篇不要求你一次用完所有技巧。推荐顺序是：先写清楚、能显示，再观察卡顿、闪烁、残影或上传限制，最后只优化真正有问题的地方。

## 为什么要性能优化

### 1. 设备资源有限

桌面电脑可以轻松处理复杂图形，但设备端更关注低功耗和稳定刷新。一个 `240 × 320` 屏幕共有 76800 个像素，如果每秒都完整重画，就会产生大量像素写入。屏幕内容看起来只是几个数字变化，实际却可能把整块背景、标题、图表和装饰线都重画了一遍。

更合理的做法是：固定内容只在整屏绘制时画，动态内容变化时只更新对应小区域。

### 2. 刷新过大会带来闪烁和延迟

如果每次 CPU 数值变化都清空整屏，用户会看到闪烁；如果一个脏区域很大，渲染器需要拆分更多条带，整帧完成时间也会变长。局部刷新可以让变化更细腻：CPU 数字变了就刷 CPU 数字区，时间变了就刷时间区，历史图变了才刷图表区。

### 3. 文件大小和内存都有上限

自定义样式文件必须控制在 16 KB 以内。把大量字体、图片数据、业务规则或采集逻辑塞进样式，会让上传失败，也会让维护变困难。样式文件应该专注于“如何显示”，数据采集和复杂计算应尽量放到电脑端或自定义数据插件中。

### 4. 复杂绘制会放大每帧成本

点阵中文、历史图表、多段进度条、网格背景都可能包含大量循环。单独看一处循环没问题，但如果每个条带、每个动态区域都重复计算，就会变成明显卡顿。优化的核心不是“少写代码”，而是减少无意义的重复工作。

## 什么时候需要优化

出现下面现象时，优先考虑性能问题：

| 现象 | 常见原因 | 优先检查 |
| --- | --- | --- |
| 数字变化时整屏闪烁 | 每次都整屏清空或脏区域过大 | `create_dirty_regions` 和 `draw_dirty` |
| 图表刷新很慢 | 单个区域面积过大或循环绘制太多点 | 拆分脏区域，使用批量接口 |
| 上传提示区域超过容量 | 脏矩形面积超过约 `240 × 40` | 拆成上下或左右多块 |
| 页面偶发残影 | 脏区域没有覆盖旧内容 | 扩大区域并先清背景 |
| 样式文件超过 16 KB | 内置数据太多 | 删除无用字模、图片或常量 |
| 同样文本重复格式化 | 多个条带重复执行同一段计算 | 使用 `prepare_frame` 缓存本帧文本 |

## 优化原则

1. 先保证正确，再优化速度。
2. 优先减少绘制面积，其次减少重复计算。
3. 优先使用 Canvas 批量接口，不要手写大量像素循环。
4. 动态区域必须覆盖旧内容的完整范围，不能只覆盖新内容。
5. 缓存只保存必要结果，不要让列表或字典无限增长。
6. 样式只做显示，不承担数据采集、网络请求和复杂业务处理。

## 手段一：把静态内容和动态内容分开

标题、分割线、固定标签通常不会每秒变化，适合放在 `draw_visible` 中；CPU 数值、时间、网络速率等会变化，适合放在 `draw_dirty` 中。

```python
class PerfDemoStyle:
    """演示静态内容和动态内容分离的屏幕样式。"""

    @staticmethod
    def create_dirty_regions():
        """声明 CPU 数值和时间两个动态刷新区域。"""
        return [
            ("cpu_value", 72, 56, 88, 24),
            ("clock", 156, 292, 76, 18),
        ]

    def draw_visible(self, canvas, snapshot):
        """绘制整屏可见内容，包含静态标签和首次动态内容。"""
        canvas.clear(BLACK)
        canvas.text(8, 12, "SYSTEM", BLUE, 2)
        canvas.text(8, 60, "CPU", GRAY)
        canvas.line(8, 88, 232, 88, DARK)
        self._draw_cpu_value(canvas, snapshot)
        self._draw_clock(canvas, snapshot)

    def draw_dirty(self, canvas, key, snapshot):
        """只重绘发生变化的动态区域。"""
        canvas.clear(BLACK)
        if key == "cpu_value":
            self._draw_cpu_value(canvas, snapshot)
        elif key == "clock":
            self._draw_clock(canvas, snapshot)
```

这样 CPU 数值变化时不会重画标题和分割线，时间变化时也不会影响 CPU 区域。

## 手段二：精准选择脏区域

`select_dirty_regions` 的作用是比较前后两帧数据，决定哪些区域需要刷新。不要简单地比较整个 `snapshot`，否则时间变化会导致所有区域都刷新。

```python
@classmethod
def select_dirty_regions(cls, previous, current):
    """按字段变化选择最小必要刷新区域。"""
    previous = previous or {}
    current = current or {}
    regions = {item[0]: item for item in cls.create_dirty_regions()}
    selected = []

    previous_cpu = previous.get("cpu", {})
    current_cpu = current.get("cpu", {})
    if previous_cpu.get("percent") != current_cpu.get("percent"):
        selected.append(regions["cpu_value"])

    if previous.get("timestamp") != current.get("timestamp"):
        selected.append(regions["clock"])

    return selected
```

如果图表使用 `cpu.history`，就只在 `history` 变化时刷新图表；如果卡片只显示 `cpu.percent`，就不要让 `history` 变化牵连 CPU 数值区。

## 手段三：把大区域拆成多块

单个脏区域不能超过条带画布容量。默认容量约等于 `240 × 40` 像素，也就是约 9600 像素。比如一个 `224 × 76` 的历史图表面积是 17024 像素，应拆成上下两块：

```python
@staticmethod
def create_dirty_regions():
    """把大图表拆成两个符合容量限制的动态区域。"""
    return [
        ("history_top", 8, 112, 224, 38),
        ("history_bottom", 8, 150, 224, 38),
    ]

def draw_dirty(self, canvas, key, snapshot):
    """上下半区复用同一个图表绘制函数。"""
    canvas.clear(BLACK)
    if key in ("history_top", "history_bottom"):
        values = snapshot.get("cpu", {}).get("history", ())
        self._draw_history(canvas, values)
```

虽然 `_draw_history` 看起来画了完整图表，但 Canvas 已经裁剪到当前条带或脏区域，只会输出可见部分。

## 手段四：绘制前先判断条带是否相交

整屏绘制时，渲染器可能按条带调用 `draw_visible`。如果当前条带在屏幕顶部，就没有必要计算底部页脚；如果当前条带在底部，也没有必要准备上方图表。

```python
@staticmethod
def _visible(canvas, top, bottom):
    """判断完整屏幕纵向区间是否与当前画布相交。"""
    return top < canvas.origin_y + canvas.height and bottom > canvas.origin_y

def draw_visible(self, canvas, snapshot):
    """只绘制与当前条带相交的内容。"""
    canvas.clear(BLACK)
    if self._visible(canvas, 0, 36):
        canvas.text(8, 12, "CPU HISTORY", BLUE, 2)
    if self._visible(canvas, 112, 188):
        self._draw_history(canvas, snapshot.get("cpu", {}).get("history", ()))
    if self._visible(canvas, 292, 310):
        self._draw_clock(canvas, snapshot)
```

这个技巧对图表、点阵中文、复杂装饰背景特别有用。

## 手段五：优先使用批量绘制接口

Canvas 提供的图表和命令接口通常比手写大量 `pixel()` 或多层循环更适合复杂图形。例如绘制历史图表时，优先使用 `draw_line_chart`：

```python
def _draw_history(self, canvas, values):
    """使用批量图表接口绘制 CPU 历史面积图。"""
    canvas.draw_line_chart({
        "x": 8,
        "y": 112,
        "width": 224,
        "height": 76,
        "maximum": 100,
        "color": BLUE,
        "filled": True,
        "regions": ((60, GREEN), (85, YELLOW), (101, BLUE)),
        "grid_step_x": 16,
        "grid_step_y": 12,
        "grid_color": GRAY,
    }, values or ())
```

如果要连续画多个矩形、线条和文本，也可以考虑 `draw_commands`，把多条绘制命令一次性交给 Canvas。这样代码更集中，也减少 Python 层反复调用的开销。

## 手段六：用 `prepare_frame` 缓存本帧结果

如果同一帧的多个条带都要使用同一段格式化文本，可以在 `prepare_frame(snapshot)` 中提前准备。它适合缓存“本帧会重复使用、计算成本不高但会被多次执行”的结果。

```python
def prepare_frame(self, snapshot):
    """在本帧绘制前准备重复使用的显示文本。"""
    snapshot = snapshot or {}
    cpu = self._number(snapshot.get("cpu", {}).get("percent"))
    memory = self._number(snapshot.get("memory", {}).get("percent"))
    self._cpu_text = "CPU {:3.0f}%".format(cpu)
    self._memory_text = "MEM {:3.0f}%".format(memory)

def _draw_summary(self, canvas):
    """绘制已经格式化好的摘要文本。"""
    canvas.text(8, 52, self._cpu_text, WHITE, 2)
    canvas.text(8, 84, self._memory_text, WHITE, 2)
```

不要把 `prepare_frame` 当成长期缓存。它只适合准备当前帧需要的结果。历史数据、网络请求结果、复杂业务状态应该由数据源提供，不应在样式里越积越多。

## 手段七：控制点阵中文和内置数据规模

中文点阵适合少量标题、标签和状态词，不适合完整文章。每增加一个汉字，样式文件都会变大；字号越大，点阵数据越多。

建议：

- 只生成本屏幕实际会出现的汉字。
- 删除示例里不用的城市、天气和提示文字。
- 中文标签尽量短，例如用“湿度”而不是“当前空气湿度”。
- 大段说明放到文档或电脑端，不放进设备样式。

如果一个天气屏幕只显示“上海、多云、湿度、风速、更新”，就不要把全国城市和全部天气描述都打包进去。

## 手段八：把数据处理移出样式文件

样式层应该读 `snapshot` 并绘制，不应该联网、读写文件或做复杂统计。比如天气接口请求、单位换算、状态归类，可以放在自定义数据插件中完成，让样式只读取已经整理好的结果：

```python
@staticmethod
def _read_weather(snapshot):
    """从扩展数据中读取已经整理好的天气对象。"""
    return (snapshot or {}).get("ext", {}).get("weather", {}) or {}

def _draw_weather(self, canvas, snapshot):
    """根据插件输出直接绘制天气信息。"""
    weather = self._read_weather(snapshot)
    city = weather.get("city", "--")
    text = weather.get("text", "等待数据")
    temperature = self._number(weather.get("temperature_c"))
    canvas.text(8, 54, city, WHITE, 2)
    canvas.text(8, 86, "{} {:.0f}C".format(text, temperature), BLUE, 2)
```

这样样式更小、更稳定，也更容易排查：插件负责数据，样式负责显示。

## 手段九：避免隐藏的性能坑

- 不要在 `draw_visible` 或 `draw_dirty` 中创建无限增长的列表。
- 不要每次绘制都重新构造庞大的字模、颜色表或命令表。
- 不要在样式中导入 `fn_canvas`，也不要判断当前是不是 C 后端。
- 不要为了“省区域”把多个无关动态内容塞进一个巨大脏区域。
- 不要把旧文字可能占据的范围排除在脏区域外，否则会残影。
- 不要在设备样式里做网络请求、文件扫描、排序大量数据等工作。

## 完整优化示例

下面示例展示一个常见仪表盘的优化结构：CPU、内存、时间分别刷新；整屏绘制时按条带相交判断；重复文本在 `prepare_frame` 中准备。

```python
"""提供性能优化结构示例的 OmniWatch 自定义样式。"""

from config import BLACK, BLUE, DARK, GRAY, GREEN, HEIGHT, WHITE, WIDTH
from styles.style_plugins import register_style


class PerfOptimizedStyle:
    """绘制采用局部刷新和本帧缓存的系统状态屏幕。"""

    name = "perf_optimized"
    zh_name = "性能优化示例"
    type = "custom"
    width = WIDTH
    height = HEIGHT
    landscape = False
    font_name = "native"

    @staticmethod
    def create_dirty_regions():
        """声明彼此独立的动态刷新区域。"""
        return [
            ("cpu", 8, 52, 224, 32),
            ("memory", 8, 96, 224, 32),
            ("clock", 148, 292, 84, 18),
        ]

    @classmethod
    def select_dirty_regions(cls, previous, current):
        """根据具体字段变化选择需要刷新的区域。"""
        previous = previous or {}
        current = current or {}
        regions = {item[0]: item for item in cls.create_dirty_regions()}
        selected = []
        if previous.get("cpu", {}).get("percent") != current.get("cpu", {}).get("percent"):
            selected.append(regions["cpu"])
        if previous.get("memory", {}).get("percent") != current.get("memory", {}).get("percent"):
            selected.append(regions["memory"])
        if previous.get("timestamp") != current.get("timestamp"):
            selected.append(regions["clock"])
        return selected

    @staticmethod
    def _number(value, default=0):
        """把未知输入安全转换为浮点数。"""
        try:
            return float(value)
        except (TypeError, ValueError):
            return float(default)

    @staticmethod
    def _visible(canvas, top, bottom):
        """判断完整屏幕纵向区间是否与当前画布相交。"""
        return top < canvas.origin_y + canvas.height and bottom > canvas.origin_y

    def prepare_frame(self, snapshot):
        """提前准备本帧多个条带可能重复使用的文本。"""
        snapshot = snapshot or {}
        cpu = max(0, min(100, self._number(snapshot.get("cpu", {}).get("percent"))))
        memory = max(0, min(100, self._number(snapshot.get("memory", {}).get("percent"))))
        timestamp = str(snapshot.get("timestamp", ""))
        self._cpu_text = "CPU {:3.0f}%".format(cpu)
        self._memory_text = "MEM {:3.0f}%".format(memory)
        self._clock_text = timestamp[11:19] if len(timestamp) >= 19 else "--:--:--"

    def _draw_cpu(self, canvas):
        """绘制 CPU 动态区域。"""
        canvas.text(8, 56, self._cpu_text, WHITE, 2)

    def _draw_memory(self, canvas):
        """绘制内存动态区域。"""
        canvas.text(8, 100, self._memory_text, GREEN, 2)

    def _draw_clock(self, canvas):
        """绘制页脚时间动态区域。"""
        canvas.text(148, 296, self._clock_text, GRAY)

    def draw_visible(self, canvas, snapshot):
        """绘制整屏内容，并跳过与当前条带无关的区域。"""
        canvas.clear(BLACK)
        if self._visible(canvas, 0, 36):
            canvas.text(8, 12, "PERF DEMO", BLUE, 2)
        if self._visible(canvas, 44, 136):
            canvas.line(8, 44, 232, 44, DARK)
            self._draw_cpu(canvas)
            self._draw_memory(canvas)
        if self._visible(canvas, 292, 310):
            self._draw_clock(canvas)

    def draw_dirty(self, canvas, key, snapshot):
        """根据区域键只重绘一块动态内容。"""
        canvas.clear(BLACK)
        if key == "cpu":
            self._draw_cpu(canvas)
        elif key == "memory":
            self._draw_memory(canvas)
        elif key == "clock":
            self._draw_clock(canvas)


def create_perf_optimized_style():
    """创建性能优化示例样式实例。"""
    return PerfOptimizedStyle()


register_style(PerfOptimizedStyle.name, create_perf_optimized_style)
```

## 优化前后的思考方式

优化前：

```python
def draw_dirty(self, canvas, key, snapshot):
    """重绘动态内容。"""
    canvas.clear(BLACK)
    self.draw_visible(canvas, snapshot)
```

这种写法虽然省事，但每个小变化都会走完整屏绘制，复杂样式很容易闪烁。

优化后：

```python
def draw_dirty(self, canvas, key, snapshot):
    """按变化区域调用对应绘制函数。"""
    canvas.clear(BLACK)
    if key == "cpu":
        self._draw_cpu(canvas)
    elif key == "clock":
        self._draw_clock(canvas)
```

优化不是把代码写得更绕，而是让每次刷新只做该做的事。

## 性能优化检查清单

上传前或遇到卡顿时，可以按这个顺序检查：

- [ ] 动态区域是否足够小，并且没有超过约 9600 像素。
- [ ] 脏区域是否覆盖旧内容可能占据的完整范围。
- [ ] `select_dirty_regions` 是否只比较必要字段。
- [ ] `draw_dirty` 是否只绘制当前 `key` 对应内容。
- [ ] 大图表是否拆成多个区域。
- [ ] 整屏绘制是否用 `_visible` 跳过无关条带。
- [ ] 图表、批量图形是否优先使用 Canvas 批量接口。
- [ ] 重复文本是否可以放到 `prepare_frame`。
- [ ] 样式文件是否仍然小于 16 KB。
- [ ] 数据采集和复杂处理是否已经放到插件或电脑端。

如果只记住一句话：性能优化的目标，是让每次刷新尽量少画、少算、少传，同时保持区域完整、代码清楚、行为可预测。
