# 第 10 课：自定义数据插件

[上一课](09-Dev模式.md) · [返回总目录](../README.md)

自定义数据插件运行在电脑端 Monitor 中，可把传感器、HTTP 接口、数据库查询或本地状态加入发送给 Pico 的 `snapshot.ext`。新版插件支持第三方 Python 库，并使用独立虚拟环境和独立子进程运行，不会与 Monitor 或其他插件的依赖版本互相影响。

## 插件目录结构

推荐把每个插件制作成一个目录：

```text
weather/
├── plugin.json
├── main.py
└── requirements.txt
```

三个文件的作用如下：

| 文件 | 是否必需 | 作用 |
| --- | --- | --- |
| `plugin.json` | 是 | 声明插件名称、数据 key、入口和采集间隔 |
| `main.py` | 是 | 实现 `collect()` 采集方法 |
| `requirements.txt` | 否 | 声明需要由 Monitor 安装的第三方库 |

插件可以直接加载目录，也可以把目录压缩成 ZIP 后分发。ZIP 内允许再套一层同名目录，但整个压缩包必须且只能包含一个 `plugin.json`。

## 编写 plugin.json

```json
{
  "protocol": 1,
  "key": "weather",
  "name": "weather",
  "zh_name": "天气数据",
  "interval": 300,
  "entry": "main.py"
}
```

字段说明：

| 字段 | 作用 |
| --- | --- |
| `protocol` | 插件协议版本，当前固定为 `1` |
| `key` | 写入 `snapshot.ext` 下的字段名 |
| `name` | 采集任务英文标识，完整任务 key 为 `custom_data.<name>` |
| `zh_name` | Windows 配置页和日志显示的中文名称 |
| `interval` | 默认采集间隔，单位为秒 |
| `entry` | 插件入口文件，默认是 `main.py`，只能位于插件根目录 |

`key` 和 `name` 只能包含英文字母、数字和下划线，不能以数字开头，最长 64 个字符，并且必须在所有插件中唯一。

## 编写 main.py

```python
#!/usr/bin/env python3
"""采集天气接口数据。"""

import requests


def collect():
    """请求天气接口并返回可 JSON 序列化的数据。"""
    response = requests.get("https://example.com/api/weather", timeout=10)
    response.raise_for_status()
    payload = response.json()
    return {
        "temperature_c": payload.get("temperature_c"),
        "status": payload.get("status", "unknown"),
    }
```

入口模块必须定义无参数的 `collect()`。返回值必须能够转换成 JSON，例如 `dict`、`list`、字符串、数字、布尔值或 `None`。不要返回文件句柄、数据库连接、对象实例和字节流。

Monitor 会在插件自己的 Python 子进程中执行 `collect()`。默认超过 10 秒仍未结束时会终止子进程，因此网络和数据库调用也应设置更短的超时时间。

## 声明第三方依赖

在 `requirements.txt` 中按标准 pip 格式声明依赖，推荐锁定版本：

```text
requests==2.32.3
```

每个插件都有自己的虚拟环境：

```text
%LOCALAPPDATA%\PicoMonitor\pluginEnvs\weather\
```

插件 A 使用 `requests==2.31.0`、插件 B 使用 `requests==2.32.3` 时不会冲突。修改 `requirements.txt` 后，管理窗口会把状态重新显示为“依赖未安装”，再次单击“安装依赖”即可更新。

依赖来自 pip 软件源。安装第三方包前请确认包名、版本和来源可信；依赖安装脚本与插件代码都具有当前 Monitor 用户能够访问的系统权限。

## Windows 加载和安装

1. 右键 OmniWatch 托盘图标，打开“自定义数据”。
2. 单击“加载 ZIP”选择 ZIP 插件包；开发时也可单击“加载目录”。
3. 选中刚导入的插件，单击“安装依赖”。
4. 等待底部日志显示“安装完成：环境就绪”。
5. 单击“测试”，确认能够看到 JSON 结果。
6. 打开“配置”，在系统采集任务中找到插件中文名称并设置采集间隔。
7. 重启 Monitor，使新增采集任务进入调度器。

Windows 安装包已经携带专用于插件的完整 Python Runtime，不需要用户另外安装 Python。Runtime 位于 OmniWatch 安装目录的 `plugin-runtime`，每个插件的虚拟环境基于它创建。

## 制作 ZIP 插件包

在 `weather` 目录的上一级执行：

```powershell
Compress-Archive -Path weather -DestinationPath weather.zip
```

不要把 `.venv`、缓存、密码文件或开发机配置放进 ZIP。Monitor 会拒绝包含路径穿越或符号链接的压缩包，也不会直接在压缩包内执行代码。

## Linux 使用

Linux 同样使用插件目录和独立虚拟环境。系统需提供 `venv` 和 `pip`：

```bash
sudo apt install python3-venv python3-pip
```

把插件目录复制到运行 Monitor 的用户数据目录后重启服务。以默认 root 服务运行时，目录通常是：

```text
/root/PicoMonitor/customData/
```

当前 Linux 服务没有图形化依赖安装窗口，可使用对应 Python 创建环境并安装依赖：

```bash
python3 -m venv /root/PicoMonitor/pluginEnvs/weather
/root/PicoMonitor/pluginEnvs/weather/bin/python -m pip install -r /root/PicoMonitor/customData/weather/requirements.txt
```

然后在 `/etc/pico-monitor.conf` 设置任务间隔：

```yaml
collection_tasks:
  intervals:
    custom_data.weather: 300
```

保存后执行：

```bash
sudo systemctl restart pico-monitor
```

## 在样式中读取数据

插件 `key` 为 `weather` 时，结果会合并到：

```python
{
    "ext": {
        "weather": {
            "temperature_c": 27.3,
            "status": "ok"
        }
    }
}
```

样式应始终使用 `.get()` 读取：

```python
ext = snapshot.get("ext", {})
weather = ext.get("weather", {})
temperature = weather.get("temperature_c")
status = weather.get("status", "unknown")
```

首次采集尚未完成、插件超时或执行失败时，对应字段可能暂时不存在。

## 常见错误

| 提示或现象 | 原因 | 处理方法 |
| --- | --- | --- |
| `环境未安装` | 尚未创建插件虚拟环境 | 选中插件并单击“安装依赖” |
| `依赖未安装` | requirements 首次加载或已经修改 | 再次单击“安装依赖” |
| `plugin.json 读取失败` | JSON 格式错误或编码异常 | 使用无 BOM UTF-8 并检查 JSON 语法 |
| `CUSTOM_DATA_NAME 重复` | 两个插件使用同一个任务名 | 修改其中一个插件的 `name` |
| `CUSTOM_DATA_KEY 重复` | 两个插件写入同一个 `ext` 字段 | 修改其中一个插件的 `key` |
| `collect 执行超过 10 秒` | 网络请求、数据库或代码阻塞 | 为外部调用设置超时并缩短单次工作量 |
| `插件子进程执行失败` | 导入失败或 collect 抛出异常 | 查看窗口底部的完整 Python 异常 |
| 选择单个 `.py` 文件无法导入 | 单文件插件格式已停止支持 | 创建含 `plugin.json` 和入口文件的标准插件目录，再加载目录或打包为 ZIP |
| pip 安装失败 | 包名、版本、网络或编译环境有问题 | 检查安装日志，优先使用提供 Windows wheel 的版本 |

建议一个插件只负责一种数据来源。天气、NAS 状态和家庭传感器分别打包，便于独立升级依赖、配置间隔和定位错误。
